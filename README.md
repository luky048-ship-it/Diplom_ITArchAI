# Стек технологий и фреймворки
В проекте должна быть реализована современная архитектура, покрывающая все слои работы с данными:

### Инфраструктура и DevOps

- **Docker / Docker Compose**  
  Локальная контейнеризация всех сервисов

- **Kubernetes (K8s)**  
  Оркестрация контейнеров (для облачного этапа)

- **Terraform**  
  Infrastructure as Code (IaC) для автоматизированного развертывания ресурсов в облаке

### Слой хранения (Storage Layer)

- **S3 Object Storage / MongoDB**  
  Data Lake (Озеро данных) для хранения сырых логов, JSON-отзывов и бэкапов

- **PostgreSQL (с расширением PostGIS)**  
  Data Warehouse (DWH) для хранения нормализованных транзакционных данных и геометрий

- **ClickHouse**  
  Слой витрин данных (Data Marts) для сверхбыстрой аналитики и колоночного хранения

### Обработка данных (Compute & ETL)

- **Apache Airflow**  
  Оркестрация пайплайнов (DAGs), управление расписанием и вызовом агентов

- **Apache Spark (PySpark)**  
  Тяжелые распределенные вычисления, очистка данных, сложные JOIN-операции между Olist и гео-полигонами

### Анализ данных и ИИ

- **Python (Pandas, GeoPandas)**  
  Исследовательский анализ, работа с пространственными данными (spatial joins)

- **LangChain / OpenAI API** (или YandexGPT / GigaChat API)  
  Фреймворки и модели для создания ИИ-агентов  
  (генерация SQL, анализ логов ошибок, генерация конфигов)

### Визуализация и Мониторинг

- **Grafana**  
  Построение дашбордов для мониторинга пайплайнов и бизнес-метрик



# Суть пректа и обобщенный план

## Суть:
Разработка MVP корпоративной платформы данных с нуля. Платформа должна автоматически адаптироваться к изменениям в структуре данных (например, при добавлении внешних геоданных) за счет интеграции ИИ-агентов. Проект реализуется в формате гибридного развертывания: сначала локальная отладка, затем миграция в облако (Yandex Cloud или Cloud.ru)

**Обобщенный план хода работы:**

1. Локальный прототип и Анализ:
    - Развертывание базовой инфраструктуры (базы данных + оркестратор) в Docker
    - Выполнение исследовательского анализа данных (EDA) датасета Olist


2. Инженерия и Агенты:
    - Настройка ETL-пайплайнов (выгрузка сырых данных, трансформация, загрузка в хранилище и витрины)
    - Написание для ИИ-агентов, которые будут генерировать конфигурации или управлять качеством данных на основе выявленных ранее паттернов.

3. Облачная миграция и PR:
    - Подготовка технического питча (с демо локальной версии) для техподдержки Yandex Cloud или Сбер (Cloud.ru).
    - Перенос архитектуры в Managed Services (K8s, Managed ClickHouse/Postgres).
    - Оценка экономической эффективности (TCO) облачного решения.

4. Оформление дипломной работы.

# Распределения ролей в компании

Для обеспечения безопасности данных и соответствия корпоративным стандартам разработана ролевая модель доступа (RBAC), включающая 16 профилей. Разделение обязанностей (Segregation of Duties) между DataOps, инженерами и аналитиками гарантирует, что сырые пространственные и транзакционные данные защищены от несанкционированного изменения, а инфраструктурные операции изолированы от бизнес-логики.

### Роли в компании для дипломной работы

**Департамент:** Data Engineering & Architecture (Разработка и Архитектура)

1. **Data Architect** (Архитектор данных)  
   - Зона ответственности: Проектирование топологии (Spark, ClickHouse, Airflow, Postgres + облачные сервисы Snowflake/BigQuery), выбор стека.  
   - Права доступа: DCL (GRANT/REVOKE) во всех базах. Доступ к мастер-веткам репозиториев (CI/CD).

2. **Core Data Engineer** (Инженер ядра данных)  
   - Зона ответственности: Написание тяжелых ETL-пайплайнов на Spark, выгрузка сырых логов Olist.  
   - Права доступа: DDL/DML в Staging / Data Lake. Чтение сырых данных. Запрет на PII без необходимости.

3. **Analytics Engineer** (Аналитический инженер)  
   - Зона ответственности: Превращение сырых таблиц в удобные витрины (Data Marts) с dbt.  
   - Права доступа: SELECT из Staging, CREATE VIEW / INSERT в аналитический слой ClickHouse.

**Департамент:** Infrastructure & Security (Инфраструктура и Безопасность)

4. **DevOps / SRE** (Инженер по надежности систем)  
   - Зона ответственности: Поддержка кластеров K8s, настройка сетей, мониторинг серверов.  
   - Права доступа: root SSH, системные утилиты. Запрет на SQL к бизнес-данным.

5. **Cloud Architect** 
   - Зона ответственности: Миграция и управление облачной инфраструктурой (AWS/GCP/Azure + PostGIS в облаке).  
   - Права доступа: Управление облачными ресурсами (IAM, VPC, storage). Мониторинг затрат (FinOps).

6. **DataSecOps / Security Engineer** (Инженер по безопасности данных)  
   - Зона ответственности: Аудит, Vault, мониторинг утечек.  
   - Права доступа: pgAudit, маскирование данных.

7. **Data Steward** (Стюард данных / Владелец качества)  
   - Зона ответственности: Метаданные, Data Catalog, разметка таблиц (геолокации, PII).  
   - Права доступа: Только метаданные и каталогизация.

**Департамент:** Data Science & ML (Машинное обучение)

8. **Geo-Data Scientist** (Гео-аналитик)  
   - Зона ответственности: GeoPandas, анализ полигонов штатов Бразилии, аномалии логистики Olist.  
   - Права доступа: SELECT исторических данных + Sandbox.

9. **ML Engineer** (Инженер машинного обучения)  
   - Зона ответственности: Вывод моделей (предсказание доставки) в продакшен.  
   - Права доступа: Feature Store + деплой микросервисов.

**Департамент:** Business Intelligence & Analytics (Бизнес-аналитика)

10. **BI Developer** (Разработчик дашбордов)  
    - Зона ответственности: Superset / Grafana / Tableau.  
    - Права доступа: SELECT только агрегированных витрин.

11. **Logistics Business Analyst** (Бизнес-аналитик логистики)  
    - Зона ответственности: Решения по маршрутам.  
    - Права доступа: Только UI BI.

12. **Marketing Analyst** (Маркетинговый аналитик)  
    - Зона ответственности: Отзывы и продажи.  
    - Права доступа: Изолированный доступ к маркетинговым таблицам.

13. **Revenue / Finance Analyst**  
    - Зона ответственности: Анализ GMV, маржи, платежей и сезонности продаж Olist.  
    - Права доступа: SELECT из витрин продаж и финансовых агрегатов.

**Департамент:** Business Data Ownership (Владение бизнес-данными)

14. **Data Product Owner** 
    - Зона ответственности: Определение требований к витринам от бизнеса, приоритизация data products.  
    - Права доступа: Только чтение витрин + утверждение изменений в Data Catalog.

**Особая категория (Системные доступы — агент-ориентированный слой)**

15. **Quality Monitoring Agent** *(улучшенная/разделённая роль)*  
    - Зона ответственности: Автоматический мониторинг качества данных, алерты на аномалии.  
    - Права доступа: Ограниченный API-токен, чтение схемы + метрик качества.

16. **Adaptation Agent**
    - Зона ответственности: Автоматическая адаптация архитектуры (создание новых витрин при изменении паттернов продаж, масштабирование в облаке).  
    - Права доступа: Генерация DDL, запуск Airflow DAG-ов, взаимодействие с Cloud Architect.
